# Auto Research · 手绘风网页 PPT 大纲

> **核心主张**：不是「交叉验证技巧」，而是 **端到端科研自动化**——从问题到可交付综述/报告，Skills 编排 + 质量门禁 + 迭代环。

## 第 1 页 · 封面

**画面（手绘）**  
笔记本封面潦草大字 **`/auto-research`**；边角小涂鸦：终端、论文堆、循环箭头（暗示 pipeline，不是单次对话）。

**页上文字**

```
/auto-research
端到端科研自动化，不是一次性写综述
Skills × Claude Code
```

**Speaker Notes**

- 主题：**Auto Research**——把「做研究」变成一条可重复、可审计、可迭代的自动化管线。
- 借 Karpathy **Capacity**：模型带宽有限，不能把设计、搜文献、写稿、审稿塞进一轮 chat；要 **分阶段加载 Skills**，Claude Code 当编排器。
- 今天展示的是我自己的 **Research Skills 世界观**（上位概念），外加一个 **Work Model 实例**——真实跑通过、约一小时量级产出的 survey。

## 第 2 页 · 手动科研 / 聊天式辅助的痛点

**画面（手绘）**  
左半边：研究生桌面（十几个标签页、PDF、笔记乱飞、时间不够）。  
右半边：把豆包/聊天助手当搜索框用（问一句、回一段），旁边标注「快，但不稳」。

**页上文字**

```
为什么慢？为什么不稳？
手动 + 聊天式辅助，都有硬伤
```

**两三点就够（页上短句）**

1. **信息爆炸，人工整理太慢**：搜文献、筛选、做笔记、搭结构，来回切换成本高。  
2. **聊天助手是“回答器”，不是“研究流程”**：它会给答案，但不会自动帮你做阶段管理和质量门禁。  
3. **缺独立复核**：同一轮对话里很容易“看起来像对的”，但引用和论证没人系统挑错。

**Speaker Notes**

- 手动流程最大问题是慢、碎、靠记忆；聊天式辅助最大问题是快但不成体系。
- 你要的不是“更会聊天的 AI”，而是“能跑完整研究工序”的工作流。
- 所以第二页是转折：从痛点进入第 3 页的端到端 pipeline。

## 第 3 页 · 正题：端到端 Auto Research 管线 ★

**画面（手绘）**  
横向或环形 **7 段 pipeline**（手绘框 + 箭头），每段一个字：设计 → 文献 → 实验 → 图表结构 → 审稿 → 修订 → 交付。中段画 **Quality Gates** 小闸门；审稿后画 **回环箭头** 指向前面的阶段（弱点路由）。

**页上文字**

```
端到端，不是单点工具
设计 → 文献 → … → 交付
不过关，不进下一阶段
```

**上位架构（页上或旁注，不写实现细节）**

| 层 | 做什么 |
|----|--------|
| **编排层** | Claude Code 按阶段 invoke 不同 Skills，真人关键节点确认 |
| **阶段层** | 研究设计 → 文献 →（可选）实验 → 图表与结构 → 同行评审 → 路由修订 → 交付 |
| **门禁层** | 文献可验证、方法说得清、结构图表过关、终审通过——**绿灯才前进** |
| **迭代层** | 审稿意见不盲目重写全文；**按问题类型回到对应阶段** 定向修，再进入下一轮评审 |

**Speaker Notes**

- 这才是文档核心：**社科/综述类研究的端到端自动化框架**。
- 交叉验证、多角色审稿、引用核验——都是这条管线上的 **组件**，不是全部故事。
- Karpathy Capacity 落点 here：每个阶段只加载当前 Skill，上下文里是 **当前阶段的 spec 与产出**，不是整库论文。
- 人在回路：自动化的是检索、整理、起草、初审；**研究问题与是否交付**仍在你手里。

## 第 4 页 · 核心思想（说人话）

**画面（手绘）**  
同一张 pipeline 上画 3 个大图标：  
1) 拆分工序（积木块）  
2) 互相挑错（多个小 agent 互评）  
3) 红绿灯门禁（不过关不往下）

**页上文字**

```
三句话：
拆开做、互相审、不过关不交付
```

**思想 + 工作流（直接讲人话）**

1. **把研究拆成阶段做**：设计、文献、写作、审稿分开跑，每步有明确产出。  
2. **让多 agent 互相 review**：不是一个模型自问自答，而是不同角色互相挑错。  
3. **不过关就回去改**：引用有问题就回文献阶段，结构有问题就回结构阶段，改完再审。

**Speaker Notes**

- 页 3 讲“流程长什么样”；页 4 讲“这流程背后的方法论”。
- 多 agent 互审是关键思想：不同视角会抓到不同错误。
- 对 PhD：核心不是写得快，是能说清楚“这份稿子被谁、按什么标准审过”。

## 第 5 页 · Work Model 示例：World Models Survey ★

**画面（手绘）**  
左侧：你输入的一个 **陌生但感兴趣** 的话题气泡——「World Models 到底是什么？」  
右侧：一叠手绘产出物（设计说明、文献矩阵、提纲、英文 survey 草稿、审稿意见）用 **约 1 小时** 时钟标注；强调 **work model**，不是 lab toy。

**页上文字**

```
Example · Work Model
话题不熟 → ~1h → 可审稿的 Survey 草稿
世界模型（World Models）文献综述
```

**示例叙事（上位，不暴露 skills 文件细节）**

| 维度 | 说了什么 |
|------|----------|
| **起点** | 对 World Models 感兴趣，但 **不是** 该领域专家；需要一张清晰的领域地图。 |
| **过程** | 走完整条 Auto Research 管线：先 **收窄范围与受众**（设计），再 **多路文献检索与合并**，再 **章节提纲与综述写作**，再 **模拟同行评审与引用审计**。 |
| **产出** | 一篇结构完整的 **英文 survey 草稿**（功能分类：Renderer / Simulator / Planner），附评审报告与待改项——**像真的在投稿前改稿**，不是 demo 段落。 |
| **性质** | **Work Model**：证明这套 Skills 在 **真实陌生话题** 上可重复；不是课堂小实验。 |

**Speaker Notes**

- 这是 `examples/world-model/` 里那套东西的 **PPT 版故事**。
- 重点：**端到端跑通过**——不是只展示「搜了 10 篇论文」，而是 design → lit → outline → draft → review 全链路有 artifact。
- 时间量级约 **一小时**（口述时可说「一个下午内的第一次 pass」——按你实际体验微调）。
- 话题选 World Models 的原因：术语混乱、综述需求真实、你自己原本不熟，**最能说明 Auto Research 的价值**。

## 第 6 页 · 这套东西是什么 / 不是什么

**画面（手绘）**  
左右对比：左 **✓** 文献综述、系统梳理、投稿前自检；右 **✗** 代替原创理论、代替敏感田野、代替真人同行评议。

**页上文字**

```
Auto Research 帮你「做研究工序」
不替你「成为研究者」
```

**四条原则（手绘清单）**

1. 每阶段确认后再前进。  
2. 论断必须能追溯来源。  
3. 工具要 **真调用**，不能假装搜过文献。  
4. 失败要透明——核验不过就停，不硬编。

**Speaker Notes**

- 收束 **边界感**：自动化的是工序与重复劳动，不是学术责任。
- 与第 2 页呼应：PhD 用 AI 的底线仍是 **来源可验证**；管线把这条写进门禁，而不是事后补救。
- 灵感除 Karpathy 外，还有业界 **分层 Skills + 迭代质检** 思路（口述可提 DeliAutoResearch 等，不必展开）。
